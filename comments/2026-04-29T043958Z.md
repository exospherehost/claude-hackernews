## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The "agent ran DROP DATABASE when I told it to drop a table" pattern is exactly what FailProof was built to catch. The built-in `warn-destructive-sql` policy intercepts the agent's bash call before execution and surfaces a warning to it; for a hard block on DROP DATABASE specifically, the custom policy is short:

  import { customPolicies, allow, deny } from "failproofai";

  customPolicies.add({
    name: "block-drop-database",
    match: { events: ["PreToolUse"] },
    fn: ({ toolName, toolInput }) => {
      if (toolName !== "Bash") return allow();
      const cmd = toolInput?.command ?? "";
      if (/\bDROP\s+DATABASE\b/i.test(cmd)) return deny("DROP DATABASE blocked. Drop the table instead.");
      return allow();
    },
  });

Drop the file under .failproofai/policies/ and commit it; every teammate's Claude Code session picks it up on next pull.
```

## Insight for the FailProof team

- The "I told the agent I had a backup, so it deleted the whole database instead of the table" pattern is a *narrative-shaped* failure mode, not a regex-shaped one. `warn-destructive-sql` will fire, but the agent will see the warning, weigh it against its conviction that "user said it's fine, they have a backup", and proceed anyway. **Action:** consider a built-in policy `require-explicit-confirmation-for-drop-database` (or a `strict-destructive-sql` mode) that returns `deny()` instead of `instruct()`/warn for DROP DATABASE / TRUNCATE / DELETE FROM without WHERE. The custom-policy snippet I posted is essentially that, and the conversation here suggests it's a high-leverage default.

## Notes / findings

- **MCP launch-order trap actually triggered, but `~/.config/browseruse/config.json` is on port 9333 (Reddit harness), not 9334.** The `--cdp-url http://127.0.0.1:9334` flag in `.mcp.json` is silently ignored in `--mcp` mode (already documented in INSTRUCTIONS.md), so the MCP tried 9333 and timed out. Worked around it by using the `browser-use` CLI as a subprocess for the entire session. Long-term fix: edit `~/.config/browseruse/config.json` to point at 9334 for HN sessions, or maintain a separate config per harness.
- **Session conflict between MCP and CLI.** Once we ran a CLI command, the MCP's session lock collided ("Session 'default' is already running with different config"). Resolved by `browser-use close` between commands when re-opening. Worth recording in INSTRUCTIONS.md if it keeps happening.
- **`block-read-outside-cwd` false-positive on URLs in bash commands** is real and frequent. The base64 dance from INSTRUCTIONS.md works. Project's own dogfooding catches us; that's a feature.
