> We built Agent Vault in response to a question that been plaguing the industry: How do we give agents secure access to services without them reading any secrets? ... ## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Egress proxy is a clean shape for "agent issues an HTTP call to a third-party API". The path it does not cover, and reasonably so given the threat model in the post, is the one most coding-agent secret leaks actually take: the agent reads a .env or `git config --list` into its own context, and the credential is in the transcript, the next prompt, or a Bash echo before any HTTP call happens. So anyone running AV alongside Claude Code or Codex still wants a tool-call-time check on file and Bash outputs to redact connection-string and JWT-shaped substrings before they reach the model. The two layers compose: AV stops the credential from leaving the network, a harness-side hook stops it from entering the model. Neither replaces the other.
```

## Insight for the FailProof team

- **Two-layer composition is the cleanest framing for FailProof vs the egress-proxy / credential-broker family** (Agent Vault, Kontext CLI, the various agent-firewall Show HNs). The pitch is not "we replace these" - it is "they handle the network egress, we handle the tool-call boundary inside the harness, both are needed because credentials leak through whichever boundary you didn't lock down". Worth turning into a short blog: "Where in the agent stack does your secret actually leak?", with a stack diagram showing prompt -> agent -> tool dispatch -> hook -> tool -> egress -> destination, and which leak class lives at each arrow. Egress proxies catch the rightmost arrow; harness hooks catch the middle.
- **The "agent reads a .env file" path is the failure mode that does not show up in any single product's threat model in this thread.** AV's design assumes the agent never has the credential. FailProof's sanitize-* family assumes the agent might read it from disk. They handle disjoint cases. We should be explicit on the docs page that FailProof's secrets story is about *agent context contamination*, not network egress, and link to AV / Kontext / similar as complements, not competitors. Cross-marketing potential is real; both projects benefit from the "two layers compose" framing.
- **`sandeepkd`'s MITM-cert-validation point applies symmetrically.** A harness-side hook also runs in-process with the agent and therefore inherits agent-process trust assumptions. It is not a SEAL boundary. Worth a docs note explaining the trust model: hooks are a *defense-in-depth* layer for a non-malicious agent that has been prompt-injected, not a containment boundary against a fully compromised agent. Same caveat AV has, applied to the layer above.
- **`dangtony98`'s "research preview, form factor unstable" framing is a useful template for our own README.** FailProof has been pushing a more polished launch tone; a "we are figuring out the right shape and welcome contribution" framing on the next major version note is more HN-native and reduces the pitch-shape friction.
