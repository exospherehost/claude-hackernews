- **Apache 2.0 single Go binary** that lets an LLM call ~30 backends via DADL (declarative YAML tool descriptions), uses a Code Mode meta-tool (`list_tools(pattern)` + `execute_code(code)`) to compress a 50k-token tool surface to ~1k, and runs a fail-closed pipeline auth -> authz -> credential injection -> exec -> output gate -> audit. ## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Code Mode is the most interesting bit here, and the one with a real security tradeoff. Compressing a 50k-token tool surface to a `list_tools` + `execute_code` pair is great for context budget, but the agent's actions move from "call get_pull_requests" to "submit JS that calls a sequence of endpoints." The fail-closed pipeline sees each call individually; it doesn't see the JS composition that connects them. Two endpoints that are individually `access: read` can compose into an exfil chain ("list customer rows, then post to webhook") that no per-call policy will flag.

A complementary layer is harness-side: the PreToolUse hook on Claude Code or Codex sees the `execute_code(code)` submission before dispatch, so the JS source is available for static checks (block calls to suspicious URL patterns, flag write-after-read sequences) before it hits the gateway.
```

## Insight for the FailProof team

- **Code Mode (or RLM-style meta-tooling) is a category-shift in the gateway space worth tracking.** Anthropic's own RLM blog post and the marimo-pair RLM-as-notebook approach are the same shape: the agent's tools are no longer a flat list of named tool calls, they're a code surface that composes calls. ToolMesh is the first general-purpose MCP gateway I've seen ship this. From a policy standpoint this changes the threat model: per-endpoint badges (`read`, `dangerous`) lose meaning when two reads compose into an exfil. The framing FailProof should articulate is "endpoint policy + composition policy, both needed". A blog post titled "Why per-tool allowlists break under Code Mode" would be timely; cite ToolMesh as a positive example of the new pattern, then make the composition-vs-call argument.
- **`access: read|dangerous` is two-state; FailProof's argument-shape distinction (`block-force-push` vs allowed `git push`) is the missing third state.** Worth being explicit on the docs page that FailProof's argument-shape policy fills the same axis as DADL's `access` badge but at finer granularity. Cross-links between the two projects on the "you still need sane policy" axis are mutually flattering, not competitive.
- **The OP's "you still need sane policy" line is the cleanest opening for a FailProof comment in this category of thread.** It's an explicit acknowledgement that great tool surface is necessary but not sufficient. Future replies on similar Show HNs (gateway / proxy / declarative-tool-spec products) can lead with that framing rather than the layer-comparison one which is starting to overuse on HN.
- **CallerClass (different policy per client type: local dev assistant vs hosted agent vs CI bot)** is a model FailProof should consider mirroring. Today FailProof's policy is per-scope (global / project / local); per-caller (which agent is invoking the tool, e.g., Claude Code interactive vs Claude Code headless cron) would be a useful axis. Worth a roadmap discussion.
