- **Submitted by:** `lahfir`, 1 day old, 93 points, 35 comments at draft time.
- **Repo:** https://github.com/lahfir/agent-desktop

OP describes agent-desktop: a cross-platform Rust CLI (~15 MB, 53 commands) that exposes a structured accessibility-tree view of native apps so an agent can act without screenshots. Loop is `snapshot --app <X> --compact` -> agent picks an `@e<N>` ref -> `click @e12` / `type @e5 "text"` / `press cmd+return` -> snapshot again. Core design choice: progressive skeleton traversal (default depth 3, named containers get refs, agent re-queries subtrees) cuts token cost 78-96% vs full-tree dumps in apps like Slack / VS Code / Notion. Author explicitly invites discussion of "implementation details that may be interesting here," and the existing comment thread is mostly LLM-detection drama and Linux/AT-SPI gaps -- the design itself has no engaged-reviewer subthread yet.

## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The progressive-skeleton traversal is the right call for token economy. The flip side of a stable @e ref model is that the agent can act on any element a snapshot returned, including ones the human never visually scanned. Once the agent has 53 commands across native apps, the cheapest place to constrain blast radius is at the perception step rather than the action step: if the agent can't snapshot Keychain Access, it can't click "Delete keychain item." For a Claude Code session shelling out to your CLI, that's a PreToolUse:Bash gate:

  customPolicies.add({
    name: "no-snapshot-sensitive-apps",
    match: { events: ["PreToolUse"] },
    fn: ({ toolName, toolInput }) => {
      if (toolName !== "Bash") return allow();
      const m = (toolInput?.command ?? "").match(/agent-desktop\s+snapshot\s+--app\s+"?([^"\s]+)/);
      if (!m) return allow();
      if (["Keychain Access","1Password","System Settings"].includes(m[1])) {
        return deny(`snapshot of "${m[1]}" blocked.`);
      }
      return allow();
    },
  });

Action-layer gates work too but require tracking which app's tree an @e ref came from across calls.
```

## Insight for the FailProof team

- **agent-desktop is a third architecture-shape worth tagging in the docs.** Existing FailProof messaging covers (a) destructive shell ops (rm -rf, force push), (b) MCP tool calls (Cordon / AgentPort space), and (c) credential brokering (Agent Vault). agent-desktop adds a fourth: *native UI control via accessibility-tree refs*. The PreToolUse:Bash hook catches it for free because the agent shells out, but the framing "gate the perception step, not the action step" is novel and specific to ref-based UI drivers (also applies to Cua Driver and the Vercel agent-browser pattern). Worth a short docs page: "Constraining native-UI agents via PreToolUse:Bash."
- **The opaque @e ref is a leverage gap.** A click takes only `@eN`; the policy can't tell from the bash call alone whether `@e12` is "OK" or "Delete." Three ways to close it: (1) gate the snapshot scope (this draft), (2) require the agent to also pass the element label as an argv comment (`agent-desktop click @e12 # "Send message"`) so the policy can grep, (3) FailProof keeps the last snapshot per session and resolves refs at click time. Option 1 is the only one that's purely operator-side; (2) needs upstream cooperation; (3) is FailProof feature work. Option 1 is the right one to lead with publicly; option 3 is a future built-in policy.
- **Comment-thread signal:** the visible reply chain is mostly about whether the OP body itself is LLM-generated (`jstanley` vouching the dead comment, `nerdsniper` arguing hybrid, `jbreckmckye` flagging Gemini watermark on the GH photo). The risk is the audience is in flame mode and may downvote even a substantive top-level comment; the disclosure line keeps the affiliation honest if it does land badly.
- **Cross-product alignment:** PR #18 (Cua Driver) takes the audit-trail-as-policy-decision-log angle. This draft takes the perception-narrowing angle. Both apply to the same physical category (native-app driving via accessibility tree) but answer different questions: PR #18 answers "how do I explain what the agent did?" and this answers "how do I prevent the agent from seeing destructive buttons in the first place?"
