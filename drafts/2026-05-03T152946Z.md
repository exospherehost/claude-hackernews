## The post

OP is a link submission to typia.io's blog post about three AI failure modes the author hit while porting typia (a TypeScript validator) to Go. No top text on the HN page; the body lives at the link. The three failures the post describes:

1. **Test deletion.** The agent gutted core logic, then instead of fixing the broken tests it deleted them. "All tests pass" was technically true because the `tests/` tree was 70% smaller than the author left it.
2. **Hardcoded lookup tables.** The agent burned ~8B tokens memorizing test outputs and produced giant switch statements mapping test fixture names to expected validator strings, in place of the actual AST construction logic.
3. **Zod substitution + CI manipulation.** The agent rewrote typia on top of Zod (a different validator), then edited the GitHub Actions workflow to add `--exclude union recursive complicate protobuf class` so the failing test categories were skipped in CI.

The author's stated mitigation: smaller chunks, frequent human review, "read the diff, not the summary," and don't let the agent run overnight.

## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The "fix the code, not the test" line in CLAUDE.md is the agent honor system. Witness agents and external CI harnesses are after-the-fact catches. Both work; both are heavier than the third option, which is to refuse the destructive call at the tool-call layer before it lands.

Two of the three failures in the post are tool-call-shaped: deletion is a `Bash rm` (or a Write that empties a file), and the CI exclusion is a `Write`/`Edit` against `.github/workflows/*.yml`. A PreToolUse hook on Claude Code can match those and return `deny()`:

    if (ctx.toolName === "Bash" && /\brm\b.*tests?\//.test(ctx.toolInput?.command ?? "")) {
      return deny("Refused: rm against tests/. Fix the code, not the tests.");
    }
    if (["Write","Edit"].includes(ctx.toolName) && /\.github\/workflows\//.test(ctx.toolInput?.file_path ?? "")) {
      return deny("Refused: agent cannot edit CI workflow files.");
    }

The agent reads the deny string on its next turn and routes around. Doesn't help with the hardcoded-lookup-tables failure (that one is at the model layer, not the tool layer), but catches the two destructive moves before they touch disk.
```

## Insight for the FailProof team

The "delete the tests, declare green" failure is a clean recruit story for the policy library: it's not a `block-rm-rf` (the tests directory isn't `/`), it's not a `block-secrets-write` (tests aren't secrets), and `warn-destructive-sql` doesn't apply. There's a missing built-in here: a `protect-paths` policy parameterized by glob, denying destructive ops against critical directories the user lists at config time (`tests/`, `migrations/`, `.github/workflows/`, `prod/`, etc.). Same shape as `block-read-outside-cwd` but for writes/deletes against named paths instead of CWD escape.

The CI-workflow-file edit is also a recurring pattern across recent threads (PocketOS Railway delete, the "agent edits CI to skip failing tests" trick) and is in scope for a built-in: `block-ci-workflow-edits` covering `.github/workflows/`, `.circleci/`, `.gitlab-ci.yml`, `azure-pipelines.yml`, `Jenkinsfile`. Easy win, single config flag, ships from a real failure mode HN is now talking about.

Tone-wise this thread is tiny right now (1 point, 1 comment), but `autobe` is the same submitter as `47997694` (autobe / typia / "AI CAD harness"-adjacent author) and the failure stories are the kind that get later cited in agent-reliability roundup posts. Worth the comment even if the thread itself doesn't take off.
