```
Respect the no-deps, system-primitives constraint. Curious where you'd draw the line on the bash tool though. Once a harness exposes direct shell, a lot of "agent destroyed my X" reports on HN are reachable from a single tool call. We've been splitting it the other way: keep harness small, push pre-tool gating (e.g. catching rm -rf outside cwd before exec) into a separate hook layer any harness can shell out to. block-rm-rf is the canonical example. Did that ever make your "not in the box" cut, or was it deliberately left to the caller?
```

**Insight for the FailProof team:**

Pu.sh's framing - "the agent loop itself is tiny, almost everything else is DX and hardening" - is a clean articulation of the architectural pitch FailProof has been making backwards. Hardening (policy, gating, sanitization) is exactly the "everything else" that doesn't belong in the harness if the harness wants to stay 400 LOC. The market signal is: independent harnesses are proliferating (Pu.sh, aloop, willhanlen.com/ai.js, plus the mainstream Claude Code / Codex / Cursor stack), and each one has to re-decide whether to bake hardening in or leave it to the caller. A separate hook-layer that any harness can shell out to is a natural fit, but only if the integration cost is genuinely low. Action items: (1) a "drop me into your harness's exec wrapper" one-liner recipe in the README, not just the Claude-Code-hooks path; (2) Show HN landing copy that explicitly addresses harness authors as a target audience, not just Claude Code users; (3) a follow-up blog where a real custom harness (Pu.sh-style or Pi-style) calls FailProof from outside Claude-Code-land - that proves the "any harness" claim. Also, Amber-chen's tool-call recording question is unanswered in the thread; that's the Agent Monitor pitch, but raising it as a top-level reply to a third party in OP's thread is high-pitch-risk and we should not do it here.
