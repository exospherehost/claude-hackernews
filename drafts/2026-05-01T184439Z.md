> In what way is it getting worse - the model's reasoning, or your own setup drifting underneath you?

**Why this user is a fit for FailProof AI:** rosenlykke is naming the exact problem the product solves: a solo founder who knows hooks/skills/CLAUDE.md matter for keeping an agent on-task, but does not have time to build and maintain that meta-layer themselves. FailProof is a curated, off-the-shelf version of that meta-layer (39 built-in policies, JS SDK for custom ones, `.failproofai/policies/` commits to the repo so teammates auto-inherit). Replying with the concrete distinction between "soft" CLAUDE.md drift and hook-time enforcement, then pointing at the install command, is direct help for their stated problem.

**My reply:**

```
Disclosure: I work on FailProof AI (https://github.com/exospherehost/failproofai), so weight that accordingly.

Your diagnosis tracks. The CLAUDE.md drift problem is real, but hooks are different in kind: CLAUDE.md is soft guidance the model can ignore mid-session, hooks fire on the actual tool call and either let it through or don't. They rot differently too. CLAUDE.md rots by getting longer and being skimmed; hooks rot by sitting unmaintained until something they should have caught slips through.

The "build the meta tools vs ship the actual project" tension is exactly the gap we built failproofai for. It is a curated set of 39 PreToolUse/PostToolUse policies you install in one command:

  npm i -g failproofai
  failproofai policies --install

That gives you block-rm-rf, block-force-push, block-push-master, block-secrets-write, block-read-outside-cwd, sanitize-api-keys, sanitize-jwt, warn-destructive-sql, warn-package-publish, and ~30 more. The point is you don't write a hook to keep the agent off main or out of your .env, you just turn it on. Custom policies are JS with allow / deny / instruct helpers when you need repo-specific rules, and dropping them in .failproofai/policies/ commits them to the repo so a teammate gets the same guardrails on next pull. Three-scope merge (global / project / local) so you can override per-project without forking.

Doesn't fix model regressions, but it takes the "did I leave a sharp edge exposed" question off the table, which sounds like the meta-layer maintenance you're describing.
```

**Insight for the FailProof team:**

- The framing "your setup is drifting" lands better on HN than "your agent is going off the rails" - the former implies operator agency and matches how this audience already talks about CLAUDE.md / skills / hooks. Worth using that vocabulary in the README and the docs landing page; the current copy leans more on the "destruction prevention" frame which is correct but lower-empathy for solo founders who are already doing safety work and just want it shrink-wrapped.
- This thread is a thinly disguised "what's your meta-layer setup" thread (see the OpenCode harness sub-thread from idempotent_, 47938318). There's a clear audience for "here is the off-the-shelf hook layer that pairs with your harness." A blog post titled something like "Hooks aren't part of the harness, they're part of the policy layer - and you can install one" would map cleanly onto this conversation.
- 47948542 (tstrimple) and 47971649 (same user, follow-up) are reporting concrete behavior regressions: same CLAUDE.md, hooks, skills, but a downgraded result on Opus 4.7. Not a FailProof problem, but an interesting signal that the user community is starting to disambiguate "model got worse" from "harness got worse" from "policy layer drifted." The product can lean into being the *stable* layer in a stack where the model and harness move under you.
- "Solo founder" was the self-description. Pricing / packaging worth thinking about: the audience reading this thread is heavy on solo + small-team users who would balk at enterprise pricing but happily install a free OSS tool that ships sane defaults. The Commons Clause license is the right call for that audience; the README could lead more aggressively with "free for individuals and teams forever" so it isn't conflated with a paid product.
