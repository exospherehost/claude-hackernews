- **Submitted by:** `frabonacci` (Francesco from Cua), 4 days old, 188 points, 41 comments at draft time.
- **Repo:** https://github.com/trycua/cua

OP describes Cua Driver: a background computer-use driver for macOS that lets an agent click, type, scroll, and read native apps without warping the human's cursor or stealing focus. Built on a SkyLight `SLEventPostToPid` trick paired with yabai's focus-without-raise pattern. Default interface is a CLI so agents like Claude Code, Hermes, etc. can shell out to it. Use cases listed: delegated demo recording, replacing Vercel's agent-browser, dev-loop QA agents, iMessage from Claude Code. OP frames the design problem as: "agents need to share hosts safely, run in the background, and avoid collisions with the human or other agents using the same machine." OP explicitly invites feedback from people building Mac automation, agent harnesses, or accessibility tooling.

## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The audit-trail "why" doesn't live in the tool-call log; it lives at the policy that fired. Tool-call logs answer what an agent did. Policy-decision logs answer which rule allowed or denied the call, plus the matched input. For Claude Code shelling out to cua-driver, that's a PreToolUse hook on Bash; the deny/allow reason is the audit row:

    customPolicies.add({
      name: "no-erp-customer-edit",
      match: { events: ["PreToolUse"] },
      fn: ({ toolName, toolInput, session }) => {
        if (toolName !== "Bash") return allow();
        const cmd = toolInput?.command ?? "";
        if (/cua-driver\s+type.*customer_email/.test(cmd)) {
          return deny(`no-erp-customer-edit; session=${session.sessionId}; cmd=${cmd}`);
        }
        return allow();
      },
    });

Compliance gets a row per decision with the rule name, the agent's input, and the session id. The tool-call log alone can't tell them why the agent didn't do something it could have.
```

## Insight for the FailProof team

- **The audit-trail-as-policy-decisions framing is a sharper sell than audit-trail-as-tool-call-log.** Most observability tools for agents (LangSmith, Logfire, Phoenix, etc.) record what tool calls happened. davey2wavey's question makes the "but why" gap explicit. FailProof has the answer baked in - every `allow(msg)`, `deny(msg)`, `instruct(msg)` carries a reason string that maps 1:1 to a compliance row. Worth a short blog post titled something like "The audit trail isn't the tool-call log" that walks through this distinction with one screenshot of what a policy-decision row looks like (without going full feature-tour). This same point lands in any compliance / SOC2 / HIPAA conversation about agent infrastructure.
- **Cua Driver is the right shape of adjacent infrastructure to namedrop alongside.** It runs *under* the agent (executes UI actions); FailProof runs *over* the agent (gates tool calls before dispatch). The two compose cleanly: a Claude Code session shells out to `cua-driver` via Bash, and a PreToolUse policy can pattern-match the cua-driver invocation by argv. Worth adding a "FailProof + Cua Driver" worked example to the docs - mac-side agent control with policy gates on iMessage / Slack / ERP web apps.
- **Pattern signal worth tracking:** "agents need to share hosts safely" (OP's phrasing) is a third architecture for this category, distinct from "MCP gateway" (Cordon, AgentPort) and "in-process hooks" (FailProof). The shared host model is where multiple agents hit the same physical machine - a forcing function for per-rule, per-session audit rows so you can attribute a destructive call to the right agent run later. FailProof's session-id propagation through `ctx.session.sessionId` already supports this; the docs don't lean on it. A doc page titled "Per-session attribution when multiple agents share a host" would land directly with this slice.
