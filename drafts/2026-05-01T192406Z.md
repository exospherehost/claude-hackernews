# Reply: AgentPort vs runtime-hook layer (Show HN: AgentPort)

**Status:** draft (pending manual post)

**HN:** thread https://news.ycombinator.com/item?id=47950752 ; parent comment https://news.ycombinator.com/item?id=47951089

**Story / OP:** Show HN: AgentPort - Open-source Security Gateway For Agents (by yakkomajuri, 6 points, 3 comments). OP frames AgentPort as a gateway that connects an agent to downstream services (Gmail, GitHub, Stripe, PostHog, Linear) with granular per-call permissions (auto-approve, ask-for-approval, never), so the agent never holds API keys and destructive ops require human confirmation. Motivated by the PocketOS incident (Cursor agent + Railway MCP destroyed a production database in 9 seconds).

**Parent comment (yakkomajuri, OP, cid=47951089):**

> OP here. Happy to answer any questions. Security for autonomous agents/claws is a deep rabbit hole that I've jumped into and I'm really interested in having open discussions about how people are approaching it.

**Why this user / thread is a fit for FailProof AI:** yakkomajuri is openly inviting design discussion on an adjacent product. AgentPort and FailProof solve the same headline problem (an agent doing something it should not) at different layers - AgentPort sits between agent and downstream API, FailProof sits inside the agent runtime as PreToolUse/PostToolUse hooks. A substantive comparison reply is exactly what the OP asked for, and the layer distinction is genuinely useful information for any HN reader trying to map this defense-in-depth space. The PocketOS reference in the OP body is the same incident FailProof's destructive-ops policies (block-rm-rf, block-secrets-write, block-force-push, block-push-master, etc.) were built to prevent on the runtime side.

**My reply:**

```
Disclosure: I work on FailProof AI (https://github.com/exospherehost/failproofai), which lives in the same problem space as AgentPort but at a different layer, so weight accordingly.

The two-vulnerabilities framing matches what we ended up with: destructive ops and credential exfiltration as the two acute failure modes worth solving first. Where we land differently is the gating layer. AgentPort gates the downstream call (Stripe, Railway, GitHub) and the agent never holds the credential. FailProof gates inside the agent runtime via PreToolUse/PostToolUse hooks for Claude Code, Codex, and Cursor Agent, so Bash, Write, Edit, and Read calls get inspected and either allowed or denied before they dispatch.

For PocketOS specifically: AgentPort would have caught the Railway delete_volume at the gateway. FailProof would have caught it if the agent had reached for a local form (rm -rf, aws rds delete-db-instance, .env read, force push to main) since those never traverse a downstream gateway. The Cursor incident is a pure-MCP path so a gateway is the right cut; on the larger surface where a lot of destruction is shell-shaped, the hook-level cut catches more.

Honest read: these are layers, not alternatives. Defense-in-depth setup has both: downstream gateway for the third-party API surface, runtime hook for the local tool surface. For everything you have not wired through your gateway (Bash, file IO, an MCP you did not gate, sub-agents that spawn shells), the runtime hook is what is left.

One question on threat model: how does AgentPort handle an agent that bypasses the gateway by writing a small script that hits the upstream API directly (curl https://api.railway.app, a vendor SDK installed via npm, a different MCP server pointed at the same provider)? That specific bypass is what PreToolUse on Bash/Write was carved out for, and it is the part of agent-tooling security I have not seen a clean answer for outside the runtime layer.
```

**Insight for the FailProof team:**

- The "two acute vulnerabilities" framing in yakkomajuri's post (destructive ops + credential exfiltration) is the cleanest version of the problem statement I have seen anyone use on HN. Worth lifting into our own positioning - we currently lead with "39 policies" which is feature-count language; "destructive ops + credential exfiltration as the two acute failure modes" is product-value language and lands harder on this audience. Consider rewriting the README's first sentence around it.
- AgentPort and FailProof are genuinely complementary. Worth a deliberate "FailProof + AgentPort" worked example on the docs landing page once both projects mature - the PocketOS case is perfect (gateway catches the API delete, hook catches the shell-form delete). If yakkomajuri is open to it, a joint blog post or doc cross-link would map the two-layer story cleanly. Slack invite + co-authored post is an obvious next move.
- The thread itself is small (6 points, 3 comments, 2 days old) so the reply will not get massive front-page exposure, but the OP is exactly the right reader and is actively inviting design conversation. Higher signal-to-noise than commenting on the saturated PocketOS thread (47911524, 1025 comments, mid-thread visibility near zero).
- "Conductor is really popular and runs agents without any sandboxing" (in OP body) is a sharp observation and an open product question for us - we have not yet evaluated coverage on Conductor specifically. Worth a one-page status on which agent harnesses we cover (Claude Code, Codex, Cursor Agent, Copilot CLI beta) and a short note on each unsupported one about why or what would be needed. Treat it as a public roadmap signal.
- The "credential exfiltration: agent never sees API keys" framing AgentPort has is notable. FailProof's analogous policies (sanitize-api-keys, sanitize-jwt, sanitize-bearer-tokens, sanitize-connection-strings, protect-env-vars) are about not letting credentials leak into agent context after they have been read; AgentPort prevents the agent from ever holding them. Different threat model (in-context vs at-rest), and worth being precise about it on our docs page so users do not confuse "we sanitize what the agent sees" with "we hide credentials from the agent entirely" - we do the former.
