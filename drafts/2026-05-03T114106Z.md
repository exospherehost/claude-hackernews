> The way I see it, having lots of agents that work together across different computers is mostly about a sync-up problem dressed up as something else. ## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

File locking is a coordination concern, but the second half of your comment (CLAUDE.md rule the agent constantly forgets) is the canonical case for moving the rule from advisory to enforced. CLAUDE.md is a hint the model can ignore mid-session; a PreToolUse hook runs in the harness before the tool call, so the agent cannot talk its way past it. Custom policy is short:

  import { customPolicies, allow, deny } from "failproofai";

  customPolicies.add({
    name: "block-child-project-writes",
    match: { events: ["PreToolUse"] },
    fn: ({ toolName, toolInput }) => {
      if (!["Write", "Edit"].includes(toolName ?? "")) return allow();
      const path = toolInput?.file_path ?? "";
      if (path.includes("/child-project/")) return deny("Parent-context agent cannot write inside child-project.");
      return allow();
    },
  });

Adjust the path predicate. Same hook fires on every machine where the agent runs, so the cross-session drift you described goes away.
```

## Insight for the FailProof team

- arctide's pattern ("CLAUDE.md rule that the agent constantly forgets, especially when switching machines") is a clean restatement of the product's own pitch in the user's own words. The instinct to write the rule down in CLAUDE.md and then watch the agent ignore it is the most-cited motivation for FailProof we have seen across HN threads — the harness has no way to enforce a markdown bullet, but a hook does. **Action:** consider a short blog post titled something like "When CLAUDE.md is not enough: turning natural-language rules into enforced policies" with a before/after diff. The "after about ten weeks I gave up and edited .syncignore" line is also gold for marketing copy — it captures the operator-resignation arc that the product short-circuits.
- The narrower failure mode here (parent repo with child project subdirs, agent strays into child paths from parent context) is *not* covered cleanly by `block-read-outside-cwd` (which is about the cwd boundary, not nested-repo inversion). Worth seeing if a `block-write-outside-allowlist` or `pathDeny`-shaped built-in would absorb this case without making users write a custom policy. The repeat we have seen of "agent writing to a path the operator's mental model said was off-limits" suggests this deserves a built-in slot.
- The Loopsy thread itself is a coordination/transport tool, orthogonal to FailProof. But "the agent ignored my CLAUDE.md instruction" recurs across nearly every Show HN of an agent harness. There is a recurring marketing rhythm here: every harness Show HN is a place where this complaint surfaces in the comments, even when the harness itself is unrelated to policy. Comment-on-sub-comment (rather than top-level reply to OP) is the lower-flag-risk move on those threads, because the sub-comment is the one that actually describes a concrete failure FailProof solves.
- Cross-machine angle is genuinely a strong selling point: drop the policy file in the repo root, both machines pick it up identically, no need to sync hook config across hosts. That argument writes itself for distributed agent setups.
