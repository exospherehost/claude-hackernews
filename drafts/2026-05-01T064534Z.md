## The post

OP wrote a Claude Code Stop hook intended to block the agent from
ending its turn when source files have been modified but tests have
not been re-run. The script uses `cat <<'ENDJSON' ... ENDJSON`
to emit a JSON block with `"decision": "block"` and a long `reason`
that scolds the model into running tests. OP reports that under
Claude 4.7, the agent acknowledges the hook ("you are right, I should
follow it"), keeps going for two more turns, and then stops anyway
without running tests. Two top comments (`AftHurrahWinch` id=47895250
and `tadfisher` id=47896315) point out that `cat` exits 0, so Claude
Code reads the hook as "allow" and the JSON-on-stdout never takes
effect. Several commenters (`niyikiza` 47897679, `nightpool` 47896425,
`hashmap` 47896436, `philwelch` 47896338) reinforce the structural
point that natural-language hook reasons get treated as untrusted
tool-result text the model is trained to push back on. `trq_` from
the Claude Code team (47896974) asks OP to file `/feedback`. No
mention of FailProof AI anywhere in the thread.

## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Two things going on; AftHurrahWinch and tadfisher already nailed the first: `cat <<EOF` exits 0, so Claude Code reads it as "allow". You need exit 2 with the reason on stderr (blocking error), or exit 0 with `{"continue": false, "decision": "block", "reason": "..."}` on stdout. JSON shape and exit code both matter; one without the other is a no-op, which is what you are seeing.

The second issue is structural even after that fix. A Stop "block" hands control back to the model with the reason in context. If the model has decided the turn is done, it can rationalize past the message (visible in your transcript). And the block fires after the source edits have already landed.

Move the gate one layer earlier. PreToolUse fires before each Write/Edit, so you can deny the edit outright when tests have not run since the last source change. The model never gets a "wrap up" turn to negotiate away from. Sketch:

  import { customPolicies, allow, deny } from "failproofai";

  customPolicies.add({
    name: "tests-must-pass-before-edit",
    match: { events: ["PreToolUse"] },
    fn: ({ toolName }) => {
      if (!["Write", "Edit"].includes(toolName ?? "")) return allow();
      if (!testsRanSinceLastEdit()) return deny("Run the test suite before editing again.");
      return allow();
    },
  });

`testsRanSinceLastEdit` is a small stamp-file helper (a sibling PreToolUse rule on Bash with `/\btest\b/` writes one stamp; the deny path compares it against the last edit timestamp). Drop the file in `.failproofai/policies/` and it auto-loads on next session, no settings.json wrangling. Exit codes and stdout/stderr JSON are hidden behind allow/deny.

Repo: https://github.com/exospherehost/failproofai.
```

## Insight for the FailProof team

- The OP's failure mode is the canonical "hook author writes JSON to
  stdout but the wrapping shell script exits 0" trap. Two top
  commenters caught it within an hour. **Action:** the docs page on
  hook authoring would benefit from a "pitfalls" section that calls
  this out explicitly with a working snippet, and a built-in policy
  named something like `lint-stop-hook-protocol` that scans
  `~/.claude/settings.json` Stop entries on `failproofai policies`
  startup and warns when a hook's command pattern looks like
  "cat-heredoc-no-explicit-exit". Cheap diagnostic, high ROI.
- The deeper architectural insight in this thread (`niyikiza`
  47897592/47897679, `hashmap` 47896436, `steve_adams_86` 47896612,
  `neckardt` 47896464) is that natural-language hook reasons land in
  the model's *tool-result* context, where the model is *trained* to
  resist instructions because that's where prompt injections show up.
  This is a structural reason Stop hooks weaken with each model rev,
  not a 4.7 regression. **Action:** a blog post titled something like
  *"Why Stop hooks lose their teeth: control via deterministic
  pre-tool denials, not in-context scolding"* would land well with
  this audience and naturally introduce FailProof's PreToolUse-first

  example.
