## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Your second question is the cleaner one. The agent didn't need write access to backups; the Railway MCP server exposed delete_volume as a single tool whose semantics included taking the backups with it. Even with tight credential scoping, the MCP tool name was the blast radius.

A PreToolUse hook is a deterministic chokepoint that fires before the MCP call dispatches. Shape:

  customPolicies.add({
    name: "block-mcp-destructive",
    match: { events: ["PreToolUse"] },
    fn: async (ctx) => {
      if (!ctx.toolName?.startsWith("mcp__")) return allow();
      if (/(delete|drop|destroy|truncate)/i.test(ctx.toolName)) {
        return deny("destructive MCP tool blocked: " + ctx.toolName);
      }
      return allow();
    },
  });

Useless if the destructive tool is named neutrally (railway_volume_remove style), but for the literal railway_delete_volume case it's a one-line stop the agent cannot route around.
```

## Insight for the FailProof team

- The two PocketOS-related drafts already in flight (PR #15 `sanitize-connection-strings` and PR #16 prod-token grep) both target the *credential-resolution* failure mode. This draft and the `comments/2026-04-29T043958Z.md` working-shape reply both target the *destructive-call* mode, but the working-shape reply gates on Bash command text (`DROP DATABASE`) while this one gates on the MCP tool name itself. That gives the team a clean two-axis story when talking about defense in depth: "what string is in the command" vs "what tool is being called". Worth a doc page or a short blog post titled something like *"Two chokepoints for agent-driven destructive calls: Bash regex vs MCP tool name"*. The audience is the same operator slice that read the original PocketOS thread.
- KingOfCoders' framing ("we fought a decade so no one has access to this") is a pure access-control / IAM critique. FailProof is the runtime layer below that, and the sibling claim is "even when the IAM layer fails or the developer cuts a corner with a too-broad MCP token, the runtime gate still fires." That sibling-claim positioning is worth a deliberate marketing line, because right now the README presents FailProof as a single layer, and a new reader could read it as competing-with-IAM rather than complementary-to-IAM. A two-layer diagram (IAM token scope on the Railway side, FailProof PreToolUse on the agent side, both fail-closed) would communicate that better.
- mgiannopoulos' Railway-specific detail (deleting a volume cascades to the volume's backups) is a real product-design note for whoever writes example configs: the MCP-tool-name regex needs to include `volume` or specific MCP server prefixes, not just SQL-shaped verbs. A future built-in policy could ship with a default deny-list keyed off well-known destructive MCP tool names per server (railway_*, supabase_*, neon_*, planetscale_*) instead of asking every user to write the regex from scratch.

The user posting will need to decide whether four touches under one operating account on one incident is too much. If yes, the right move is to pick one of the four to actually post and let the others sit as reference.
