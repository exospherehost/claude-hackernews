**My reply:**

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The "strict permissions nerfed autonomy" tradeoff is real - the permission system is a coarse tool/pattern allowlist, so you toggle between "Bash off" and "Bash on every prod-shaped command." A PreToolUse hook that returns allow / deny / instruct per call collapses that dial: the borderline calls return instruct (the tool runs, but extra context lands in the agent's next turn so it can self-correct), which is the slot the allow/deny binary doesn't have. Concretely:

if (cmd.includes("git push --force")) return deny("force-push blocked - open a fresh branch and rebase");
return allow();

Container-per-issue handles host blast radius. PreToolUse handles the calls that look the same inside Docker as outside - force-push to your real GitLab, for example.
```

**Insight for the FailProof team:**

The OP's framing - "I set strict permissions, which nerfed Claude's autonomy" - is the cleanest one-line articulation of why coarse allowlists fail and per-tool-call policy returns earn their keep that we have seen on HN. It is also notable that the OP reached for Docker-per-issue, not hooks, when "agent unchecked" scared them. That ordering matters for marketing: container isolation is a familiar primitive, so it is the first thing engineers reach for; PreToolUse hooks are a Claude-Code-specific feature people learn about second. A landing-page section / blog post titled something like "We tried strict permissions, then containers, then hooks - here is what each one solves" would map directly onto the path users like t0mas88 actually walk. Concrete integration angle: a Smithy-compatible Docker image flavor (e.g. a `claude-with-failproof` base) that bakes FailProof hook config into `~/.claude/settings.json` inside the container, so the container-isolation user picks up intent-gating without touching their orchestrator.
