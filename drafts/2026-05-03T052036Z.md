**My reply:**

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Blast-radius story holds cleanly while --writable-mounts is off. The moment that flag flips for an agent that needs to edit a real project, the sandbox boundary stops at the mount point: an `rm -rf /workspace/*` from inside the VM lands on host files. That is the slice where the agent is doing exactly what its user asked, so the catch has to happen at the tool-call layer (PreToolUse on Bash/Write before exec), not at the VM perimeter. So the two stack rather than overlap: VM for everything outside the mount, intent-gating for ops that cross back.

Curious how you are thinking about this seam: does SmolVM want to own the in-VM policy (tags on the sandbox declaring allowed tool calls), or do you expect users to layer that via the agent's own hook config inside the guest?
```

**Insight for the FailProof team:**

SmolVM's README explicitly documents `--writable-mounts` as a flag the user *opts into* when they want the sandbox to edit host files — and that opt-in is the moment the isolation guarantee is intentionally broken. There is a clean technical-content blog post in this: "where sandbox boundaries don't help, and what does." The framing isn't "FailProof vs sandboxes" — it's "here is the specific seam where intent-gating earns its keep even when you have hardware isolation." The CelestoAI / SmolVM crowd is the natural audience: they already believe in defence in depth, they just stop at the VM line. This thread also surfaces an integration angle worth thinking about: an `smolvm claude start` flavour that boots a sandbox with FailProof policies pre-installed in the guest's `~/.claude/settings.json` — a one-liner that gets you both blast-radius reduction (VM) and intent-gating (PreToolUse), without users having to wire either side themselves.
