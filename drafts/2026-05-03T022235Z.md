## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The bypass worked because the matcher only looked at the binary name. Once a general-purpose interpreter is allowlisted, every restricted operation is reachable through it; tightening the allowlist just shifts the bypass to whichever interpreter is left. The fix is to classify by what the call actually tries to do, not which binary runs it. Claude Code's PreToolUse hook gives you the full Bash payload before execution, so a custom matcher can scan the script body for the operations you wanted to block:

  import { customPolicies, allow, deny } from "failproofai";

  customPolicies.add({
    name: "block-git-push-via-any-shell",
    match: { events: ["PreToolUse"] },
    fn: ({ toolName, toolInput }) => {
      if (toolName !== "Bash") return allow();
      const cmd = toolInput?.command ?? "";
      if (/(?:^|[\s;|&])git\s+push\b/.test(cmd)) return deny("git push detected (regardless of outer interpreter)");
      return allow();
    },
  });

Same shape catches rm -rf, curl | sh, etc., regardless of whether bash, powershell, or python is the outer call.
```

## Insight for the FailProof team

- The `claude-code` allowlist/blocklist is matched on the *first* token of the bash invocation, so any wrapper command (`bash -c "..."`, `pwsh -c "..."`, `python -c "..."`, `node -e "..."`, here-docs, `eval $(...)`) launders the inner action past the gate. eterm's failure case is the canonical version: agent gets `git` denied, so it writes the operation into a powershell script and runs the script. **Action:** worth shipping a built-in policy named something like `block-script-shell-launders` that detects "outer interpreter + inner blocked verb" patterns out of the box (parametrize on the inner-verb list). The custom-policy snippet above is the regex-shaped first cut; an AST or small parser of the bash command would be more robust against `e\val`, base64-then-decode, IFS games, etc.
- This thread is a useful pointer to a non-obvious risk surface for the marketing copy: the existing claude-code permission model is a *binary-name allowlist*, not a *capability allowlist*. The framing "GTFOBins-style bypass works against your agent's permission system too" is a clean way to explain why per-binary allow/deny is structurally insufficient. Could become a short blog post: *"claude-code's allowlist has a GTFOBins problem"* with the snippet above as the takeaway.
