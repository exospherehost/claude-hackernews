## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Interesting layer choice. We landed on the agent-hook layer (Claude Code's PreToolUse/PostToolUse and the equivalents in Codex and the Agents SDK), and the tradeoff cuts in a specific direction: a proxy at the MCP boundary catches calls across many clients with one config but misses any bash/file/edit operation that wasn't routed through MCP; an agent-side hook catches every tool call in one agent but needs per-agent integration. Worth being explicit about which one the threat model actually needs.

On "write-detection without enumerating every dangerous tool name": what worked for us is matching on argument shape rather than tool-name allowlists - e.g., an `rm -rf` pattern on any bash-input field, independent of which tool emitted it. Holds up when a new MCP server ships a wrapper that renames the verb.
```

## Insight for the FailProof team

- **The MCP-gateway-vs-agent-hook layer choice is the most concise way to position FailProof against this category of tooling.** Cordon, AgentPort, BetterClaw, and Burrow are all proxy/gateway-shaped at the MCP layer; FailProof is hook-shaped at the agent layer. The tradeoff is genuine, not marketing - a one-paragraph "when do you want each" framing on the docs landing page would help users self-route. Suggest a section in the README: "Where FailProof sits vs. an MCP proxy". Same content can land as an HN/Twitter post.
- **`block-rm-rf` is the strongest lead-in for layer-choice conversations** because it shows the "argument shape, not tool name" approach in one line. The OP literally asked how to do write-detection without enumerating tool names; that's a stable framing FailProof can return to in marketing copy.
- **HITL synchronous approvals are the one thing FailProof does not do today.** `instruct(msg)` and `deny(msg)` are agent-side responses, not human-side approval prompts. Worth deciding deliberately whether to add an `approve(msg)` primitive (probably backed by a Slack/terminal/HTTP webhook) or to leave the human-in-the-loop space to gateway-shaped tools and stay in the autonomous-policy lane. Either is defensible; ambiguity is not.
