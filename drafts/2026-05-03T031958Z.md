## The post

OP submits OMAR (Open Multi-Agent Runtime), a TUI dashboard for managing a "large swarm" of coding agents. Two co-founders (Karim and Shaokai) describe the workflow pain that motivated it: with multiple Claude Code windows running in parallel, the human spends most of the session ctrl-tabbing between terminals to review what each agent is doing. OMAR collapses that into a single dashboard and adds hierarchical agent management ("agents managing agents, like how companies work"). They explicitly solicit feedback: "If you find OMAR interesting, give it a try. We would love to hear from you."

## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The "100 agents in parallel" framing makes the review-per-tool-call model break. With one Claude Code window I can eyeball every bash call; with 100 I can't, and approving everything blind is what the prod-DB-deletion threads warn about. The fix that scales is policy-as-code at the PreToolUse hook so the human-in-the-loop becomes a function-in-the-loop, applied uniformly across every agent in the swarm:

    customPolicies.add({
      name: "no-prod-writes",
      match: { events: ["PreToolUse"] },
      fn: ({toolName, toolInput}) => {
        if (toolName !== "Bash") return allow();
        return /\bDROP\s+(DATABASE|TABLE)\b/i.test(toolInput?.command ?? "")
          ? deny("destructive SQL blocked at the policy layer")
          : allow();
      },
    });

Curious whether OMAR gives sub-agents an isolated session/cwd, or whether they share parent state. The answer drives where the policy boundary should sit.
```

## Insight for the FailProof team

- The OMAR launch is a clean signal that the "swarm of agents" framing is becoming the natural production shape, not a research-paper shape. That changes the question FailProof answers from "how do I avoid one bad command" to "how do I keep a fan-out of N untrusted agents from each independently doing the bad command." The policy-as-code-shared-via-config story fits this exactly: drop a file in `~/.failproofai/policies/`, every spawned agent picks it up, no per-agent configuration needed. Worth a blog post titled something like "When you have 100 agents, your CLAUDE.md is not enough" that walks through this pattern with OMAR as the concrete example.
- The "agents managing agents hierarchically, like how companies work" framing has a sharp policy implication: parent agents become the approval surface for child agents. Today that's modeled as in-context prompting ("supervisor agent, please verify before merging"). FailProof's `instruct()` return type is a closer fit: a parent policy can inject context into a child agent's tool call ("you are a sub-agent of <parent-id>, your DDL writes require parent confirmation") without depending on the model to remember a prompt-time instruction. Worth prototyping a `failproofai-omar-bridge` example that shows this composition.
- Two-author Show HN with explicit "we would love feedback" framing is exactly the kind of thread where adjacent-product comments land well IF they engage on the design rather than pitch features. The "review-per-tool-call breaks at N=100" framing here is genuine on-topic engagement; the policy snippet is the substrate, not the subject. If the comment lands, watch for follow-ups asking how PreToolUse hooks compose with sub-agent spawn boundaries - that's the next-blog-post seed.
- Cross-reference with the AgentPort thread (47950752, PR #11): yakkomajuri framed the same problem one layer down (gateway permissioning per integration). OMAR runs at the orchestration layer, AgentPort at the integration layer, FailProof at the per-tool-call layer. Three slices of the same risk surface; comparison post would be useful product-marketing copy.
