## Existing comments worth knowing

- `natloz` (1d): "I feel this is not the trajectory we want to go with automation. Perhaps better checks and balances within the automation, or 'thresholds' that trip breakers would be a good approach?" - argues for automated policy + thresholds, not 100% manual.
- `hasperdi` (2d): "Numerous prompting will cause prompt fatigue, similar to pressing yes on dialog boxes." - classic click-through-fatigue concern.
- `embedding-shape` (2d): "if I had to approve each command of the agent, that'd remove 90% of the benefits... wrap the agent in a way so it cannot destroy stuff in the first place. The substantive sub-thread is the design tradeoff: 100% manual approval (Fewshell's stance) vs codex-as-root + sandbox (embedding-shape) vs automated thresholds (natloz). Nobody has named the third option: programmable per-tool policies. That is the FailProof shape.

## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The "no auto-approve" stance is principled - it solves the "I forgot I left it on" failure mode cleanly. But 100% manual approval hits the same wall browsers hit with permission prompts a decade ago (the prompt-fatigue point above), and codex-as-root sandboxing only works while nothing important is inside the box. There's a middle the thread keeps not naming: programmable per-tool policies. A hook fires on every tool call and returns one of allow / deny / instruct(msg). Read-only stuff goes through silently; only patterns flagged at write time prompt or block. For shell:

  customPolicies.add({
    name: "shell-rm-rf-gate",
    match: { events: ["PreToolUse"] },
    fn: async (ctx) => {
      if (ctx.toolName !== "Bash") return allow();
      const cmd = ctx.toolInput?.command ?? "";
      if (/\brm\s+-rf?\s+/.test(cmd)) return deny("rm -rf gated");
      return allow();
    },
  });

That's where most teams land once approval-on-everything triggers click-through fatigue.
```
