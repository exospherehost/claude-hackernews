**My reply:**

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

Your "firewall layer to limit what tool requests the agent can even make" is what Claude Code's PreToolUse hook surface is for. The hook sits between the harness and tool dispatch, and gets to allow or deny per call before anything reaches the inner sandbox. It works the same whether the harness is in or out, because the decision is just a function on tool name plus inputs:

    customPolicies.add({
      name: "no-rm-rf",
      match: { events: ["PreToolUse"] },
      fn: ({toolName, toolInput}) => {
        if (toolName !== "Bash") return allow();
        return /\brm\s+-rf\b/.test(toolInput?.command ?? "")
          ? deny("rm -rf blocked at the policy layer")
          : allow();
      },
    });

Doesn't replace the inner sandbox. It just keeps the requests that should never get there from being dispatched at all.
```

**Insight for the FailProof team:**

Mendral's post is a useful reference frame for explaining where FailProof sits in a "harness outside the sandbox" architecture.
