## My reply

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The proxy-vs-harness-hook seam is the substantive question. AegisProxy sits between the agent and MCP servers, so it sees every MCP tool call but not the harness's built-in tools. In Claude Code, Bash, Edit, Write aren't MCP; they bypass the MCP wire entirely, and most canonical destructive failures (the Replit DB wipe being one) happen through that path. A proxy at the MCP boundary cannot see those commands. A PreToolUse gate inside the harness sees the literal toolInput before the call resolves and can deny on it.

Two layers, different failure surfaces. Cross-agent / cross-MCP-server policy belongs at the proxy. Per-machine "this command is destructive right now" belongs at the harness hook.
```

## Insight for the FailProof team

- **Architectural framing post.** The OP's framing ("we need an infrastructure layer for agents") is correct in spirit but underspecified about *where* that layer goes. The proxy-vs-harness-hook split is the real architectural seam, and the right framing is "MCP-only agents (Claude Desktop, plain MCP clients) get the proxy; harness-runtime agents (Claude Code, Codex, Cursor agent mode, Aider) need an in-harness hook because Bash / Edit / Write don't traverse MCP." A blog post titled something like *"Where does the agent control plane live: at the MCP wire, or inside the harness?"* would slot directly into this conversation and any next "agent did X destructive thing" front-page thread. We can also point readers to it from any future reply on this seam without re-deriving the argument.
- **Competing-product map gets fuller.** This thread surfaces *two* known competitors I had not previously catalogued:
  - **AegisProxy** (`aegisproxy.com`, OP `aegisproxy`) -- MCP-layer security proxy, targeting Claude Desktop and MCP servers. ~80% local sub-millisecond inspection per their pitch. Same enforcement-point philosophy as `Cordon` (47941823) and `AgentPort` (47950752, also OSS) but specifically MCP-wire.
  - **Sentinel Core** (`datawizual.github.io`, commenter `EldorZ`) -- pre-commit-layer enforcement (git-side blocker, not runtime). Different enforcement point entirely. Worth tracking as a non-overlapping layer; pre-commit catches "what gets into the repo" while FailProof catches "what gets executed during the agent session."
- **OP signal.** The OP's own framing ("Is this a real problem? Are we at the 'this needs infrastructure' stage?") is a genuine "the framing isn't crisp yet" signal. The category isn't named; everyone's reaching for analogies (IAM, VPCs, audit logs from the 2010 cloud era). Whichever vendor lands the canonical name and architecture-diagram first owns it. FailProof's specific advantage is that the harness-hook surface is already officially supported by Claude Code (and now Codex), so we are not inventing a new boundary -- we are filling the policy layer Anthropic and OpenAI both shipped without filling.
- **Saturation check.** Thread is 13 days old, 1 point, 5 comments. Mid-thread visibility is near-zero, but OP gets notified of new replies, and EldorZ's reply (11 days after OP) shows competing-product pitches still get engagement here. The reply form is open. Cost of posting is low; expected return is one engaged reader (OP) + a clean architectural artifact for the repo.
