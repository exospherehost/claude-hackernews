**My reply:**

```
(disclosure: I work on FailProof AI: https://github.com/exospherehost/failproofai)

The snapshottable-paused VM-per-agent design gives you a clean cost story and host blast-radius. What it doesn't gate, and probably shouldn't, is what the agent does inside its own VM with the credentials you mounted in: a git token, an npm token, a cloud profile, an MCP server URL. A prompt-injection or a stop-hook-ignored hallucination can have an agent force-push to a real upstream from inside the guest, drop a database through an outbound MCP, or destroy the only branch with work in progress. The VM perimeter is silent on those - they look like normal egress. Intent-gating at the agent's own PreToolUse hook (Claude Code, Codex Agents SDK) sits inside the guest and stops the call before dispatch; block-force-push is the most boring example. Stacks under Bhatti rather than competing.
```

**Insight for the FailProof team:**

Bhatti is a useful counterpart to the Mendral "harness outside the sandbox" piece (PR #17, id=47990675): Mendral argues the harness moves out, Bhatti argues the agent moves into a VM-per-session that pauses when idle. Both designs leave the same intent-gating gap inside the agent's reach: VMs and outside-harnesses isolate blast radius, but neither bounds what the agent decides to do with the credentials it was handed. There is a clear blog-post-shaped piece here titled something like "The credentials your agent has are the thing your sandbox cannot protect" - audience is exactly the Bhatti / SmolVM / Mendral overlap (people who already believe in defence in depth and stop at the perimeter). One concrete integration angle worth filing: a Bhatti rootfs image that boots with a curated `~/.claude/settings.json` containing FailProof's hook config pre-installed, so `bhatti pull failproofai/agent-rootfs` gets you sandbox + intent-gating in one pull. Would also nudge users toward a default "creds in env, policies on top" posture instead of plain `--writable-mounts` followed by a wide-open agent.
